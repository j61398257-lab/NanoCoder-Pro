# NanoCoder 学习指南

> 目标：能读懂每个文件在做什么、为什么这样设计，以及如何从零指挥 AI 把它重建出来。不要求自己手写代码。

---

## 第一课：AI Agent 到底是什么

**普通聊天机器人** = 你问它答，它只能说话，不能动手。  
**AI Agent** = 你给它任务，它会**思考 → 动手 → 看结果 → 再思考**，循环直到完成，最后告诉你结果。

NanoCoder 的核心循环只有一句话：

> **用户说话 → 大模型想 → 要不要调工具？→ 调工具 → 看结果 → 再想 → … → 直接回复用户**

**关键词，看一眼就够**

| 词 | 大白话 |
|----|--------|
| LLM | 大脑，就是 GPT / Kimi / DeepSeek |
| Tool（工具） | 手，读文件、写文件、跑命令 |
| Function Calling | 大模型告诉你「我要调哪个工具、参数是什么」的标准格式 |
| Token | 模型处理文字的计量单位，对话有上限（比如 128K） |
| System Prompt | 写给大模型的「工作手册」，告诉它是谁、能干什么、规矩是什么 |

---

## 第二课：项目全景地图

整个项目 1300 行，每个文件一句话：

```
nanocoder/
├── config.py        「入职表」—— 读取 API Key、模型名等配置
├── prompt.py        「工作手册」—— 告诉大模型它是谁、能干什么
├── llm.py           「电话线」—— 跟大模型通信，发消息、收回复
├── agent.py         「项目经理」—— 核心循环：想 → 干 → 想 → 干 → 回复  ★最重要
├── context.py       「记忆管家」—— 对话太长时压缩，防止超限
├── session.py       「存档系统」—— 把对话存到磁盘，下次继续
├── cli.py           「前台接待」—— 终端界面，接收输入、显示输出
└── tools/
    ├── base.py      「工具模板」—— 所有工具的统一接口
    ├── read.py      读文件
    ├── write.py     写文件
    ├── edit.py      改文件（搜索替换）★
    ├── bash.py      跑命令 ★
    ├── grep.py      搜内容
    ├── glob_tool.py 找文件
    └── agent.py     派子代理干活 ★
```

**数据怎么流动：**  
用户说话 → `cli.py` 接收 → 交给 `agent.py` → `agent.py` 通过 `llm.py` 问大模型 → 大模型说「帮我读 main.py」→ `agent.py` 叫 `tools/read.py` 去读 → 结果送回大模型 → 大模型说「帮我改一行」→ 叫 `tools/edit.py` 改 → … → 大模型说「搞定了」→ `cli.py` 显示给你

## 第三课：逐个模块说人话

### config.py —— 最简单，先热热身

它干的事：读 `.env` 文件 + 系统环境变量，把 API Key、模型名整理好，传给其他模块用。

读它时关注两点：
- `_load_dotenv()`：自己 16 行手写，不依赖第三方库，且**不覆盖**已有环境变量
- `from_env()` 里 Key 的优先级：专用变量优先，通用变量兜底

---

### tools/base.py —— 工具系统的「合约」

定义所有工具必须长什么样。只有三样东西：
- **name**：工具叫什么名字（大模型按名字叫它）
- **description**：工具能干什么（大模型读这个决定要不要用）
- **parameters**：需要什么参数（大模型按这个格式传参数）

`schema()` 方法把这三样打包成 OpenAI 规定的格式。`execute()` 执行后永远返回一段文字，方便塞回对话历史。

---

### 七个工具

**★ edit.py —— 最精妙的设计**

改文件不用行号。大模型告诉它「找到这段文字，替换成这段文字」。  
为什么不用行号？多步编辑时行号会漂移，而精确字符串匹配永远可靠。  
安全机制：`old_string` 必须在文件里**刚好出现 1 次**，0 次或多次都会报错，让大模型补充更多上下文再试。

**★ bash.py —— 最危险的工具**

能跑任意命令，所以必须有安全检查。  
- 9 条正则规则拦截危险命令（`rm -rf /`、fork bomb、`curl | bash` 等）  
- 用模块级变量记住 `cd`，下次执行时在正确目录里  
- 输出超长时保留头和尾（中间通常是重复内容）

**★ tools/agent.py —— 子代理**

主代理可以「派一个分身」去独立完成子任务。分身有自己的对话记忆，不会污染主代理的上下文。防止无限套娃：分身的工具列表里没有 `agent` 工具本身。

**read / write / glob / grep**：直白的文件操作和搜索，看一遍即可。

---

### llm.py —— 跟大模型打电话

三件事：
1. **流式响应**：大模型一个字一个字蹦出来，`on_token` 回调实时显示
2. **tool_calls 累积**：工具名和参数是分多个数据块发来的，需要拼起来再解析
3. **重试机制**：限速或超时时，自动等一会儿再试（最多 3 次，每次等更久）

---

### agent.py —— 整个项目的心脏

核心循环五步：
1. 用户消息加入对话历史
2. 检查记忆太长了吗？太长就压缩
3. 把「工作手册 + 对话历史」发给大模型
4. 大模型回复：纯文字 → 任务完成返回；工具调用 → 执行 → 结果加入历史 → 回第 3 步
5. 最多循环 50 次，防死循环

一个细节：System Prompt 每次调用时拼在消息列表最前面，但不存进 `self.messages`——这样 `/reset` 清空历史时不需要特别处理它。

多个工具调用时用线程池并行执行，更快。

---

### context.py —— 记忆管家

对话时间长了，工具输出多了，Token 会超限。三层应对，像医院分诊：

- **用了 50%**：截短工具输出，只保留头 3 行和尾 3 行
- **用了 70%**：让大模型自己总结前面的对话，只保留最近 8 条原始消息
- **用了 90%**：紧急，只留摘要 + 最近 4 条

---

### prompt.py —— 工作手册

34 行生成 System Prompt，每次启动时动态填入：当前工作目录、系统信息、可用工具列表、8 条行为规则（先读后改、验证结果等）。新增工具时提示词自动更新，不用手动改。

---

### session.py —— 存档系统

把对话存成 `~/.nanocoder/sessions/*.json`。下次用 `-r session_id` 恢复。`list_sessions` 预览每个存档的第一条用户消息。

---

### cli.py —— 前台接待

三种运行方式：`nanocoder`（交互）、`nanocoder -p "问题"`（单次）、`nanocoder -r id`（恢复会话）。

REPL 里内置 7 个命令：`/help`、`/reset`、`/model`、`/tokens`、`/compact`、`/save`、`/sessions`。

## 第四课：从零造一个 Coding Agent，按这个顺序

```
第 1 步：调通一次 API，能聊天就行
    ↓
第 2 步：加 tools，让大模型能说「我要调工具」，先接一个假工具测试
    ↓
第 3 步：把工具结果塞回 messages，循环直到大模型不再调工具
         ← 这就是最小可用 Agent，对应 llm.py + agent.py
    ↓
第 4 步：接真实工具：读文件 → 写文件 → 改文件 → 跑命令
         ← 对应 tools/ 目录
    ↓
第 5 步：加安全拦截、流式输出、上下文压缩
         ← 对应 bash.py 安全检查 + context.py
    ↓
第 6 步：加终端 REPL、命令系统、会话保存
         ← 对应 cli.py + session.py
```

第 1–3 步是核心，完成后就有一个能用的 Agent。后面都是锦上添花。

## 第五课：NanoCoder 从 Claude Code 提炼了什么

| 设计模式 | 大白话 | 对应文件 |
|---------|--------|---------|
| 搜索替换编辑 | 改代码不用行号，用唯一字符串匹配 | `tools/edit.py` |
| 并行工具执行 | 同时干多件事 | `agent.py` |
| 三层上下文压缩 | 记忆太多就压缩 | `context.py` |
| 子代理 | 派分身干子任务 | `tools/agent.py` |
| 危险命令拦截 | 防止搞坏电脑 | `tools/bash.py` |
| 会话持久化 | 存档/读档 | `session.py` |
| 动态提示词 | 工作手册随环境变化 | `prompt.py` |

## 推荐阅读顺序

1. `config.py`（热身，最简单）
2. `tools/base.py` → `tools/read.py`（理解工具接口）
3. `tools/edit.py`（理解最精妙的设计）
4. `llm.py`（理解怎么和大模型通信）
5. `agent.py` ← **最核心，反复读**
6. `tools/bash.py`（理解安全设计）
7. `context.py`（理解记忆压缩）
8. `prompt.py`、`session.py`（很短，顺一遍）
9. `cli.py`（界面层，最后看）

读某个文件时，配合 `tests/` 里对应的测试用例一起看，测试用例会告诉你「这个模块的期望行为是什么」。运行测试：

```powershell
py -3.11 -m pytest tests/ -v
```
